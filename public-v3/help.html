GETTING STARTED

What Icarus is & a tour of the console

Icarus is a career-automation bot for Umamusume. It drives the real Steam game client through the game's own network API, then runs an entire trainee career for you turn by turn: it reads the game state, scores every legal action, picks the best one, executes it against the live game, and adapts to whatever comes back — low energy, bad mood, a forced race, a skill it can afford. You point it at a trainee, a support deck and your settings, press RUN CAREER, and walk away.

The default scenario is Trackblazer (“Start of the Climax”): a 78-turn career whose climax/finale races land on turns 74, 76 and 78. The bot understands this scenario's rules — global stat caps of 1200, fatigue that stops after Late December, and the two summer training windows (turns 37–40 and 61–64) — and tunes its decisions around them.

The per-turn run loop

Every turn follows the same five-step cycle. You can watch each step happen live in the ACTION LOG and DECISION REASONING panels.

1 · ConnectThe bot reads the current game state over the API — your trainee's stats, HP/energy, mood, fans, skill points, the support cards on each facility, and any pending event or forced race.
2 · DecideThe Trackblazer decision engine scores every option (train Speed/Stamina/Power/Guts/Wit, race, rest, or recreation) against your stat targets, the mood/energy “waterfall”, rainbow/bond bonuses and the failure-chance ceiling, then picks the single best action. The reasoning is written to the Decision Reasoning card for that turn.
3 · ExecuteThe chosen action is sent to the live game: it trains the facility, enters the race, rests, or takes recreation — and along the way it buys/uses shop items (hammers, megaphones, energy/mood items, etc.) and purchases skills when it has the points and a good candidate.
4 · AdaptIt re-reads the result and corrects course — restoring TP before training, resting when energy is low, treating a bad condition at the infirmary, retrying a lost race with clocks/carats (if enabled), and re-planning around the upcoming race schedule.
5 · FinishAt turn 78 the career finalizes. With LOOP on, the bot deletes the finished career and immediately starts the next one; otherwise it stops and waits for you.

The status rail: brand, theme switcher & logout

Running down the left edge of every page is the STATUS RAIL — the vertical strip with the ICARUS wordmark at the top and a LOGOUT button at the bottom. Two of its controls are easy to miss:

ICARUS logo — accent theme switcherThe ICARUS brand mark at the top of the rail is also the theme cycler (its tooltip reads “Click to switch accent theme”). Each click steps through six accent colors — amber (the default), cyan, emerald, violet, rose and ice — recoloring the whole console (highlights, bars, buttons). Your choice is saved to the browser (localStorage key icarus_theme), so it survives reloads and applies on every page. To use it, just click the logo repeatedly until you land on the color you like; to reset to the default, keep clicking until you reach amber.
LOGOUTThe LOGOUT button at the bottom of the rail drops your Steam session: it POSTs to /api/logout and then reloads the page, which brings the Steam login overlay back up. Use it when you need to re-authenticate or switch Steam accounts. It does not stop a running career on its own — stop the bot first with STOP if a run is in progress.

The dashboard's three columns + MONITOR dock

The dashboard body is laid out in three columns under the resource rail and tab strip:

First run & login

Icarus controls the real Steam game client — it doesn't simulate anything. So before the bot can do anything it needs a working game install and a captured Steam login. This page walks the first launch end to end.

Prerequisites

Umamusume installed via Steam, run onceInstall the game through Steam and launch it at least once normally, so the account is set up and the game files are in place. Icarus drives this exact client.
Python 3.10+ and Node.js 18+ with npmRequired to run the bot. From the project folder run pip install -r requirements.txt and npm install once, then start the server with python main.py and open the dashboard at http://127.0.0.1:1616.
RUN AS ADMINISTRATOR (WINDOWS)

The Steam login capture attaches to the running game process to read its credentials. On Windows that requires elevation — start the bot from an Administrator terminal. If login capture keeps failing or never finds the game, this is almost always the cause.

Set the userdata path (DIAG / AI)

Before logging in, choose where your data lives so it survives version upgrades. Open the DIAG / AI tab and click USERDATA FOLDER. The modal shows your current folder, how it was resolved, and the pointer file location.

  1. Type a stable path outside the build folder into SET A STABLE FOLDER (e.g. C:\Umamusume API Bot\Icarus_userdata).
  2. Leave Also copy current settings/presets/auth on — it does a non-destructive copy of your existing data into the new folder.
  3. Click APPLY PATH, then restart the bot so the new location takes effect everywhere.

If you're not ready, click I'LL HANDLE IT LATER. The same walkthrough auto-pops on first run; an amber warning appears if you're still on the in-build fallback folder (your settings would be lost on the next upgrade).

Steam login (the login overlay)

When the backend is reachable but no account session exists, Icarus shows a full-screen STEAM LOGIN overlay (it also appears as a gate on the ACCOUNTS-related flow). Credentials are sent only to your local bot, never anywhere else — the overlay footer states this explicitly. The login is a two-step flow: first credentials, then (only if Steam Guard demands it) a one-time code.

  1. Enter your USERNAME and PASSWORD and click LOGIN. The button changes to WORKING… while the request is in flight. You can also press Enter in either field to submit.
  2. If the bot replies that two-factor is required (needs_2fa), the username/password fields are hidden and a STEAM GUARD CODE field appears, the button switches to VALIDATE, and a message prompts you to check your authenticator. Enter the code (the field auto-uppercases) and click VALIDATE.
  3. The bot launches/attaches to the game, captures fresh in-game auth, and loads your account. On success the page reloads into the live dashboard.
YOUR USERNAME & PASSWORD ARE REMEMBERED

On a successful login the overlay caches your credentials in the browser's localStorage (saved_username / saved_password) so the fields auto-fill next time the overlay appears. These never leave your machine — they are only ever POSTed to your local bot. To clear them, sign out of the browser profile or clear site data for 127.0.0.1:1616.

FAILED LOGINS CAN TRIGGER A COOLDOWN

If Steam rate-limits repeated attempts, the bot returns a cooldown_seconds value and the login button is disabled while it counts down (e.g. LOGIN · 30s / VALIDATE · 30s). Wait for the countdown to finish — the button re-enables automatically — rather than hammering it. Other failures show the bot's error message in red above the button (wrong password, network error reaching the bot, etc.).

KNOWN: CAPTURE MAY NEED TWO PASSES

On some setups the very first login is rejected at the Steam Guard step and a second attempt succeeds. If the first try bounces, simply log in again — and make sure the bot is running elevated.

Quick start — install to a running career

The fastest path from a fresh install to a bot training a career. Do steps 1–5 once; steps 6–10 each time you start a run.

  1. Install Umamusume via Steam and launch it once normally so your account is set up.
  2. In the Icarus project folder, install dependencies: pip install -r requirements.txt and npm install.
  3. Start the server (Windows: from an Administrator terminal) with python main.py, then open http://127.0.0.1:1616.
  4. On the DIAG / AI tab, click USERDATA FOLDER, set a stable path outside the build folder, click APPLY PATH, and restart the bot.
  5. When the STEAM LOGIN overlay appears, enter your Steam username/password (and Steam Guard code if prompted) and finish login so your account loads.
  6. Go to SETUP and pick your TRAINEE (use LIBRARY to browse owned characters).
  7. Pick your parents / a GUEST parent, build or pick the support DECK, and choose a FRIEND support.
  8. Optionally choose a SETTINGS PRESET (or tune Training / Racing / Skills via the settings modals).
  9. Return to DASHBOARD and press RUN CAREER.
  10. Watch it go in the ACTION LOG, DECISION REASONING, and the MONITOR dock. Use PAUSE / STOP any time.
TIP

If RUN CAREER flashes PICK A TRAINEE IN SETUP, BUILD/PICK A DECK, or PICK A FRIEND SUPPORT, finish that choice on the SETUP page first. To run careers back-to-back, turn on LOOP in the rail before pressing run.

OFFLINE BANNER

An OFFLINE bar means the dashboard can't reach the backend. Make sure python main.py is running, then reload — the page goes live automatically on the next poll.

Getting started

The userdata folder

The userdata folder is the single place Icarus keeps everything you've personalized. Pointing it at a stable location outside the build folder is the one setup step that protects your configuration across version upgrades.

What it stores

Settingssettings.json — your dashboard/engine settings.
Presetsdata/presets/ — your saved Training/Racing/Skills setting presets.
Accountsaccounts.json — the multi-account roster used by the Accounts page and the supervisor.
Steam authauth/<profile>/steam_token.txt and auth/<profile>/auth_config.json — your captured Steam session, so you don't have to re-login every upgrade.
Picker selectionactive_selection.json — your last-chosen trainee/deck/friend/parents, so Setup and the dashboard remember your pick between restarts.

Why it survives upgrades

The build folder gets overwritten when you install a new version. Anything stored there — settings, presets, your Steam auth — would be wiped. The userdata folder lives outside the build, and Icarus records your chosen location in a cross-version pointer file at ~/.icarus/userdata_pointer.json (older ~/.sweepycl paths are still read). A fresh install of a future version reads that pointer and automatically picks up where you left off — no re-login, no lost presets. The USERDATA FOLDER modal (DIAG / AI) shows the current folder, how it was resolved, and warns in amber if you're still on the temporary in-build fallback.

Resolution order, highest priority first: an ICARUS_USERDATA_DIR environment variable; the saved pointer file; an Icarus_userdata folder sitting beside the build; and finally the build folder itself as a last-resort fallback. The multi-account supervisor is launched with the same userdata folder, so all your bots share one roster.

Where game-save data is read from

Important distinction: the userdata folder holds Icarus's configuration — it does not contain your Umamusume save. Your trainee stats, owned cards, currencies and career state live on the game's servers. Icarus reads them live over the game API after you log in (you'll see them in the resource pills and vitals), and writes finished-career records into its own history/logs under the runtime/userdata area. So the game is always the source of truth for save data; the userdata folder is only Icarus's own settings and auth.

Moving or fixing the path

  1. Open DIAG / AI → USERDATA FOLDER.
  2. Enter the new path under SET A STABLE FOLDER.
  3. Keep Also copy current settings/presets/auth on to bring your existing files along (it never overwrites files already at the destination).
  4. Click APPLY PATH, then restart Icarus.
TIP

If the modal warns the resolved path is a legacy location or the in-build fallback, set a fresh Icarus path here once with migrate enabled — you'll get a clean, upgrade-proof folder going forward.

SETUP

Trainee & support deck

The Setup page is where you assemble everything a career needs before you hit run: the trainee you're raising, two parents (inherited factors / sparks), a six-card support deck, an optional friend support card borrowed from another player, and the Settings Preset that bundles all of your tuning. Everything on this page is loaded live from your game account the moment you finish the in-game login, so there is no manual data entry.

HOW DATA GETS HERE

When Setup opens it calls /api/session. If the bot console is still waiting for you to enter the game menu you'll see a NOT LOGGED IN card; if the Python server isn't running at all you'll see OFFLINE with a reminder to run python main.py. Once you're in the game, your trainees, parents, decks and owned cards populate automatically — no refresh needed.

The five team slots

At the top of Setup, a row of slots shows your current picks. Each slot fills in as you click items in the Library below. The slots are:

TRAINEEThe horse girl you're raising. Highlighted with the amber accent because it's required — nothing solves or runs without it. Shows the scenario name (Trackblazer) as its sub-label.
PARENT 1Your first inherited parent (one of your own trained characters). Provides sparks / inheritance factors.
PARENT 2Your second parent. This slot accepts either a second one of your own parents or a guest parent borrowed from someone you follow. When a guest is picked the slot relabels itself to PARENT 2 · GUEST.
DECK · SLOT 4Your six support cards, shown as six thumbnails. Clicking this slot opens the Build Custom Deck modal. The slot number reflects the in-game deck slot the chosen deck lives in.
FRIENDAn optional seventh support borrowed from another trainer's lent card. Leave empty to run with just your own six.

Every pick you make is saved to the server immediately (via /api/selection) and persisted to disk, so your selection survives a server restart and is restored the next time you open Setup.

The Library and its six tabs

The Library panel is the catalog of everything on your account. A search box at the top right (search library) filters the current tab by name as you type. Each tab header shows a live count of how many items it holds. The tabs are:

TRAINEESEvery trainee on your account. Click one to select it (it lands in the TRAINEE slot, replaces any previous pick, and the card gets an amber check). This is loaded directly in the initial session payload.
PARENTSYour own trained parents, available for inheritance. This tab has a full filter bar (see below) and spark tooltips on hover. Click up to two — they fill PARENT 1 then PARENT 2. Click a selected one again to deselect it. Loaded in the initial session payload.
GUEST PARENTSParents borrowed from people you follow (your follow list / succession rentals). This tab does not load with the session — it loads on demand. You'll see a LOAD GUEST PARENTS button; click it to fetch the list (it calls /api/career/guest_parents). Selecting one places it in PARENT 2. Spark tooltips work here too.
DECKSYour saved in-game support-card decks, each shown as a card-strip with its slot number. Above the list sits the always-on DECK BONUSES panel. Click a deck to select all six of its cards at once. Hover any deck for a per-card scoring tooltip.
FRIEND SUPPORTSSupport cards lent by other players. Like Guest Parents, this loads on demand via a LOAD FRIEND SUPPORTS button (calls /api/career/friends) and is blocked while a career is actively running. Each card shows its type and limit-break. Click to pick one as your FRIEND.
OWNED CARDSA read-only browse of every support card you own (type, limit-break, rarity). Useful reference when hand-building a deck; cards here aren't directly clickable into a slot.
WHY GUEST PARENTS & FRIENDS LOAD SEPARATELY

Both come from a different game call (the pre-single-mode / follow-list payload) than your own roster, so they're fetched only when you ask. The first load is cached for the session; later it serves instantly from cache unless you force a refresh.

Selecting a trainee

  1. Open the TRAINEES tab.
  2. (Optional) type into search library to narrow the grid.
  3. Click the trainee. Her card gets an amber border + check, and she fills the TRAINEE slot.

The trainee drives almost everything downstream: the deck score is computed against her growth type, and the Smart Race Solver uses her aptitudes to pick races. Pick her first.

Building the six-card support deck

You have two paths to a deck:

  • Pick a saved deck — in the DECKS tab, click any deck. All six of its cards drop into the DECK slot at once.
  • Build your own — click the DECK slot, the BUILD CUSTOM DECK button, or the BUILD DECK button inside the Deck Bonuses panel. This opens the Build Custom Deck modal where you hand-pick six cards.

The DECK BONUSES panel (always visible above the deck list, and as a tooltip when you hover a deck) scores the selected deck out of 10 with a verdict, and breaks down the combined stat bonuses, type counts, total limit-break, type-match %, bonus strength and LB density. If a trainee is selected it scores against her growth type; otherwise it tells you to pick a trainee to refine the type match. If you see a "master data not loaded" message, run a master-data sync so card effects can be read.

Reading the Deck Bonuses panel & hover tooltip

The always-on DECK BONUSES panel and the tooltip you get by hovering a deck card both score the same way, pulling effect data live from /api/supports/details at each card's current limit-break and level. Read them like this:

Score out of 10 + verdictA single overall grade (e.g. 7.4/10) with a one-word colour-coded verdict beside it. Green = strong, amber = fair, red = weak. The panel shows the score big; the hover tooltip shows it compact next to the deck name and its slot number.
Type chips (×count)One coloured chip per support type present in the deck — Speed, Stamina, Power, Guts, Wit, Friend — each with a ×N count of how many cards of that type the deck holds. Use it to spot an unbalanced deck at a glance.
Combined effect totalsThe panel lists every stat/effect the deck grants, summed across all six cards and sorted strongest-first (e.g. Friendship Bonus +XX%, Training Effectiveness +XX%). These are the totals you actually get in-game at the cards' current LB.
Per-card ordered effectsThe hover tooltip additionally shows each card individually — its type, rarity, limit-break (LB) and level — with its top few ordered effects listed below. Cards with nothing yet read "no bonuses at this level" (raise the card's level / LB in-game).
Footer metrics — Total LB · Type match · Bonus · LB densityA summary line: Total LB = sum of every card's limit-break; Type match % = how well the deck's types align with the selected trainee's growth type (needs a trainee picked); Bonus % = overall effect strength; LB density % = how limit-broken the deck is on average. Higher is better across the board.
SCORE IS TRAINEE-AWARE

With a trainee selected, the panel reads "Scored vs <trainee>" and folds her growth type into the type-match figure. With no trainee it scores the raw deck and reminds you to pick one to refine the type match — so for an accurate verdict, choose your trainee before judging a deck.

Picking a friend support

In the FRIEND SUPPORTS tab (after clicking LOAD FRIEND SUPPORTS), click a card to borrow it as your seventh support; click it again to clear it.

IMPORTANT — KEYED BY LENDER, NOT CARD

A friend support is identified by its lender's viewer_id combined with the card id — not by the card id alone. Many different players lend the same card (e.g. several copies of "Fine Motion"). If selection were keyed by card alone, clicking one would highlight every copy and could send a borrowed card that duplicates one already in your deck, which the game rejects (result code 2511). Keying by lender means only the exact copy you clicked is selected. If you don't see the friend you expect, make sure you clicked the specific lender's tile.

Picking parents (own + own vs own + guest)

The career start always uses exactly two parents, and there are two modes the UI chooses for you automatically based on what you pick:

Two of your own (own_own)Pick two parents from the PARENTS tab. They fill PARENT 1 (own[0]) and PARENT 2 (own[1]). This is the default when no guest is selected.
Own + guest (own_guest)Pick one of your own parents (PARENT 1 = own[0]) plus one guest from the GUEST PARENTS tab. The guest takes PARENT 2, which relabels to PARENT 2 · GUEST. At run time the guest is sent as a rental (its lender viewer_id, instance and card id) rather than as a second owned parent.

You can hold at most two own parents and at most one guest. Selecting a guest does not consume an own-parent slot beyond PARENT 2 — it simply occupies that second slot. Clicking a selected parent or guest again removes it.

PARENT SPARK TOOLTIPS

Hover any parent or guest tile to see its full inheritance tree: the parent plus its two grandparents, each with G1/G2/G3 win counts and colour-coded factor rows (blue = stat, pink = aptitude, green = unique, gold = scenario) with star ratings. Use it to judge spark quality at a glance before committing.

The Parents filter bar

The PARENTS tab has a dedicated filter bar (its settings are remembered in your browser) for digging through a large roster. A live count below it reads N/M parents so you can see how aggressively the filters are cutting the list:

Search parent / sparkFree-text box labelled Search parent / spark…. It matches against both the parent's name and any factor (spark) name anywhere in its lineage, so you can type a stat or skill name to find every parent that carries it.
Category chipsSTAT / APTITUDE / UNIQUE / SKILL / RACE / SCENARIO toggle chips — keep only parents that carry a factor in the chosen category. Click to toggle each on/off; selected chips turn amber. They stack (multiple chips = parents matching any of them).
Any factorA dropdown of every distinct factor present across your parents, grouped by category. Pick one to filter to parents that have it.
Minimum stars (≥ 1★ … ≥ 9★)A nine-step dropdown — ≥ 1★ through ≥ 9★ — setting the minimum star total a parent must have for the chosen factor. Defaults to ≥ 1★ (no restriction). Only meaningful once a specific factor is selected in Any factor.

Cleaning up throwaway parents (Age window + PREVIEW CLEANUP)

If you mass-create parents just to harvest sparks, the PARENTS filter bar includes a safe two-step bulk-delete tool. It is a destructive action wrapped in a mandatory dry-run, so you always see exactly what will go before anything is removed:

  1. Pick an age window in the dropdown (options run from 1h up to 7d). Until you choose one, the button is disabled and the hint reads "Select age window to preview cleanup." Once chosen, the hint updates to how many recent parents fall inside the window and reminds you that selected/active parents are protected.
  2. Click PREVIEW CLEANUP. This runs a dry-run first (POST /api/parents/remove-recent with dry_run:true) and shows you a sample list (up to 10 names + a "…and N more" tally) of the parents created inside that window that would be deleted.
  3. Confirm in the dialog. Only then does it issue the real delete (dry_run:false). On success you're told how many were deleted and prompted to sync to refresh the list.
PROTECTED & IRREVERSIBLE

Parents you've selected (your current PARENT 1 / PARENT 2) and active parents are excluded automatically and can never be caught by cleanup. Everything else inside the age window is fair game, and the deletion cannot be undone — the dry-run preview exists precisely so you can check the list before confirming.

How guest / followed parents work + Refresh

Guest parents are real trained characters lent by people on your follow list (not support-card summaries). They load only when you press LOAD GUEST PARENTS. After the first load the list is cached for the session, so reopening the tab is instant.

  • To pull a fresh list (e.g. you've followed new people, or you want different rentals offered), use the REFRESH button shown on the tab. This re-calls the game with force_refresh set, bypassing the cache.
  • Refresh can also pass an exclude list, so re-rolling won't keep re-offering the same lenders you've already seen.
  • If the call fails you'll get an error panel with its own REFRESH to retry; if your follow list simply has no rentals you'll see "No guest parents found on your follow list."

Settings Presets — the saved bundle of everything

A Settings Preset is a named bundle that holds your full tuning: Training Settings, Racing Settings, Configure Skills, Scenario Overrides, the Smart Race Solver config, and your team selection (trainee, deck, friend, parents, guest). Switching presets is how you keep, say, a "Mile sprinter" config separate from a "Long-distance stayer" one. The preset block is in the right-hand controls column:

Preset dropdownLists all your presets. The one shown is the one the buttons below act on.
NEWPrompts for a name, creates an empty preset, and immediately makes it the active preset. Use this to branch a fresh config.
SAVEWrites the current Setup state — including your live team selection — onto the preset named in the dropdown. Use it after tweaking settings or changing picks so they persist.
LOADMakes the dropdown's preset the active preset and reloads the page so every setting and selection re-reads from it. This is how you switch between presets.
DELDeletes the selected preset (after a confirm). Cannot be undone.
ACTIVE PRESET MATTERS

Skill config, solver config and the saved race list are all stored per-preset. Whichever preset is active is the one a career actually uses. LOAD sets the active pointer (and reloads); NEW sets it to the freshly-created one. If your settings don't seem to take effect, confirm the right preset is active.

The RACE SCHEDULE calendar at the bottom shows Junior / Classic / Senior years with each scheduled race banner. Under the Smart Race Solver this is filled by the solver; in Manual mode you click a date to pick a race yourself.

The Aptitude Threshold grade picker (Solver)

Inside SMART RACE SOLVER SETTINGS, the APTITUDE THRESHOLD section is not a single fixed "C" value — it's an eight-grade button row. The buttons are S A B C D E F G, and the selected one is highlighted (default C). It sets the minimum aptitude — in both distance AND surface — a race must meet to be considered eligible by the solver. To use it, click the grade you want as your floor: raise it (e.g. to B or A) to make the solver pickier and only enter races your trainee is genuinely suited to; lower it (e.g. to D) to open up more of the calendar for a weak or specially-routed character.

The Manual / per-aptitude grade overrides (Solver)

The APTITUDES section above the threshold lets you override the trainee's aptitudes that the solver plans against, per distance and surface. Each aptitude is shown as a row reading Base · Manual Start · Sparks + · Solver Final, so you can see exactly how the final planning grade is built up:

BaseThe trainee's stock aptitude for that distance/surface, from her character data.
Manual StartThe value the solver actually starts from. By default it equals Base, but you can override it by clicking a grade button on that row — useful when you intend to push an aptitude up via inheritance and want the solver to plan as if it's already there.
Sparks +The estimated parent-spark contribution, added on top of Manual Start.
Solver FinalThe resulting grade (Manual Start + Sparks) that is actually sent to the route solver. This is what the APTITUDE THRESHOLD above is compared against.

To override a row, click the grade button (one of the same eight S…G buttons) on that aptitude; click the Base button on the row to clear your override and snap Manual Start back to the trainee's stock value. The RESET APTITUDES button in the modal footer clears every manual override at once.

How to set up a run — full walkthrough

  1. Start the server (python main.py) and complete the in-game login until the Library populates (no NOT LOGGED IN / OFFLINE card).
  2. Pick (or create + LOAD) the Settings Preset you want this run to use, and confirm its Training / Racing / Skills / Scenario settings are how you want them.
  3. Open the TRAINEES tab and click your trainee. (Do this before solving — the solver and deck score depend on her.)
  4. Open PARENTS and pick two of your own (own_own), or pick one own parent here plus one in GUEST PARENTS (own_guest). Use the filter bar and spark tooltips to choose good factors. Press LOAD GUEST PARENTS / REFRESH if you want fresh rentals.
  5. Choose your support deck: click a saved deck in DECKS, or use BUILD CUSTOM DECK. Check the DECK BONUSES score against your trainee.
  6. (Optional) load FRIEND SUPPORTS and pick one borrowed card for the FRIEND slot.
  7. Decide race planning: leave SMART RACE SOLVER on, then SYNC DATA → SOLVE SMART (review the plan) → APPLY SMART to write it to the active preset. Or switch to MANUAL and pick races on the calendar.
  8. Press SAVE on the preset so your selection and any tweaks are persisted onto it.
  9. Verify the five top slots are filled (trainee + both parents + deck, friend optional), then start the career from the navbar Run control.
TIP

If you ever get a guest parent silently ignored, or a friend that won't apply, re-check the two pitfalls above: a guest only takes effect when PARENT 1 is also one of your own (own_guest needs own[0]), and a friend must be picked by its specific lender tile.

SETUP

Smart Race Solver

The Smart Race Solver is Icarus's race-route planner. Instead of you hand-picking every race, the solver looks at the trainee's aptitudes, the whole 78-turn Trackblazer calendar, and a set of scoring weights, then chooses the set of voluntary races to enter across the entire career that maximises your objective (fans, set-bonus epithets, or a blend). It runs as a true optimiser: the backend uses a SciPy MILP (mixed-integer linear program), with a beam-search fallback if MILP is unavailable. You see this status on the TRACKBLAZER card in Setup as smart_backend: MILP, milp_available: yes, beam_fallback: ready (verified in public-v3/setup.js).

The solver only plans your optional racing. Mandatory in-game races (the ones the game forces) and the climax/finale races at turns 74/76/78 always run regardless of the plan. The engine then layers a prediction “double-star” gate at execution time, so a planned race the bot judges unwinnable is backed out of and replaced or trained over (verified in career_bot/scenarios/mant_trackblazer.py).

By default the solver doesn't just plan once — it can re-solve the remaining schedule as the run unfolds. These three nested toggles control that (verified in modals.js).

MATCHER CAVEAT

The epithet matcher uses condition text from the race cache, not the game's real completion logic. Exact race-name conditions match precisely, but broad phrases (Dirt, Mile, Sprint, G1…) just flag matching rows. A forced epithet being “satisfied” is not a guarantee the in-game title actually completes (documented in docs/smart-race-solver-settings.md).

Smart vs Manual selection

MANUAL mode lets you hand-pick exactly which races to enter; it is authoritative — the engine will not train over, substitute, or prediction-gate your manual picks (verified in mant_trackblazer.py: _is_manual_mode short-circuits the marquee guard, irregular-training hijack, low-energy block, and the prediction gate). Manual selections round-trip with the preset: they save to extra_race_list with extra_race_list_source: "manual" and reload on the next session. (The RACE SCHEDULE card to the right of the solver shows the booked calendar as a game-style grid for reference; the run follows whatever race list is saved on the active preset.)

MANUAL WINS TIES

Defense-in-depth in the backend (main.py): if a run is launched as “smart” but the active preset was saved as manual with a non-empty race list, the bot respects your saved manual choice rather than silently letting the solver take over. So a stale browser state can't downgrade a hand-built schedule into dirt races you never picked.

  1. On the Trackblazer card, click MANUAL (or SMART RACE SOLVER to go back to the optimiser).
  2. For Smart: pick a trainee, set aptitudes/eligibility/weights in SOLVER SETTINGS, click SOLVE SMART to preview, then APPLY SMART to commit. Or just leave auto-solve before run ticked.
  3. For Manual: build your race list, which persists as source: manual; the run will enter exactly those races.

Marquee set-races run automatically

Independently of Smart vs Manual, the engine has a must-run marquee guard (verified force_marquee_races, default ON, in mant_trackblazer.py). The fixed prestige set-races — Takarazuka Kinen, Arima Kinen, Japan Cup, Tenno Sho (Spring/Autumn), the Triple-Crown races (Satsuki Sho, Tokyo Yushun, Kikuka Sho, Oka Sho, Oaks, Shuka Sho), Queen Elizabeth II Cup, NHK Mile, Yasuda Kinen, Mile Championship, Victoria Mile, Takamatsunomiya Kinen, Sprinters Stakes, Champions Cup — are too valuable to skip. In smart mode, if the game offers one this turn, the bot races it even over a smaller planned race or a training turn. The guard is still gated by your Max Streak and by energy: it requires vital ≥ marquee_min_vital (default 30) or an owned energy item, except year-end marquees (e.g. Arima), which are always allowed since the year ends right after. In manual mode this guard is off — your list is final.

SETUP

Skill strategy

Icarus spends your trainee's skill points (SP) for you. Every turn after a result it looks at the skills your supports and trainee have made available (the in-game “skill tips”), scores them, and buys the best ones it can afford — so you never have to open the skill menu yourself. This page is the Configure Skills modal (open it from Setup; the modal title is SKILLS) plus the engine behind it. Every control below was verified in public-v3/modals.js and career_bot/skills.py.

THE 30-SECOND VERSION

Pick a running style in Racing Settings (skills inherit it), leave Enable Skill Point Check ON, set a sensible Skill Point Threshold (default 888), and leave Stop After Recommended Skills OFF. That is the configuration that reliably buys good skills. If you want hand-picked skills only, use Manual Skill Tiers at the bottom.

How automatic buying works (the flow)

Each turn the buyer (the SkillBuyer class in career_bot/skills.py) runs this gauntlet, and it skips buying entirely unless it passes:

  1. Master switch. Enable Skill Point Check must be ON, or nothing is bought (skills.py buy()).
  2. Threshold gate. Your current SP must be above the Skill Point Threshold (or above the hard-coded hoarding ceiling of 1500 SP, which always triggers a spend). Below threshold, the turn is skipped with reason threshold.
  3. Pre-finals override. From turn 73 onward (the run-up to the Twinkle Star Climax finale races at 74/76/78) the threshold is ignored and any SP > 0 is dumped on the best skills, so the trainee enters the finals fully kitted.
  4. Candidate scoring. Available tips are resolved to real skills, filtered (owned, blacklist, wrong-style, off-distance, skipped colors), and ranked by true value-per-SP (in-game evaluation points ÷ SP cost × your aptitude grade) with a hand-tuned heuristic as the tiebreaker.
  5. Affordability fill. It buys the top picks until SP runs out, capped to one green skill per purchase by default.

It never re-buys a skill the trainee already owns. The game only stores a partial, rotating view of owned skills, which used to make older versions re-buy the same skill repeatedly; Icarus now keeps a career-long owned set so this can't happen (_acquired_skill_ids, skills.py _candidates).

Under SKILL PLANS are three switchable tabs — click one to reveal its panel. They control when the bot is allowed to spend, and all three reuse the same weighted scoring/purchase rules described above.

Pre-FinalsAn extra plan that fires in the run-up to the finals. Its panel has a single Enable Plan toggle (config key pre_finals_enabled). Default OFF.
Career CompleteAn extra plan that fires at career completion. Its panel also has a single Enable Plan toggle (config key career_complete_enabled). Default OFF.

Both Pre-Finals and Career Complete default OFF on purpose. The engine already force-spends every remaining SP from turn 73 onward regardless of these toggles (see step 3 above), so a trainee never carries SP past the last races it can affect. Leave them off unless you have a specific reason; they exist for fine-grained control and are distinct from that automatic late-game dump.

THE #1 “BOT ISN'T BUYING SKILLS” FIX

If your trainee finishes a career with a big pile of unspent SP, check Stop After Recommended Skills first — set it OFF. When ON, the gate discards every candidate on any turn where none of the available skills happens to be a top-tier “recommended” skill, instead of stopping only after the recommended ones are owned. The result is hoarded SP and almost no purchases. This was the confirmed root cause in normal careers; the default is now OFF (config_store.py and skills.py). Second thing to check: your Skill Point Threshold isn't set absurdly high (e.g. above what the trainee ever earns). Third: a stale server — a running bot loads its config once, so after changing skill settings you must fully restart the bot (a browser refresh is not enough).

Group Cards (Sirius / Throne)

Two of the game's support cards are Group cards rather than ordinary single-character cards: Team Sirius (support id 30081) and Heirs to the Throne (support id 30067). Instead of one fixed partner, a Group card carries a roster of characters who unlock as group outings (a special recreation option, command 390) across the career. Each outing builds bond with a whole pod of characters at once and feeds the card's multi-step story chain. This section of Training Settings tells Icarus how aggressively to spend those outings. If neither card is in your deck, every control here is inert and you can ignore the whole section. WHERE IT LIVES

Open the gear / Training Settings modal and scroll to the GROUP CARDS (SIRIUS / THRONE) section near the bottom (just under DISTANCE). It is part of the same preset you save with the rest of Training Settings.

What a group outing actually does

A group outing is the recreation tile that the card injects on certain turns. Firing it (command 390 with a select_id naming which character to visit) advances that character's bond and pushes the card's story forward. Icarus tracks each character's state from the game payload as a pair: is_outing (1 = the outing has unlocked and is sticky) and story_step (0 = not yet done, 1 = done) — see _char_outing_state (mant.py:275-282). It only ever fires an outing that is unlocked and not yet done, so it never wastes a turn re-visiting a finished character.

Team Sirius and Heirs to the Throne also have a small number of card-itself story steps layered on top of the character outings. The Throne card has 2 card steps; Team Sirius has 1 (GROUP_CARD_OUTING_MAX = 2, mant.py:26). These are the payoff steps of each card's story.

Item Strategy

Icarus runs a full shop economy on top of training. Every turn it visits the MANT shop, decides what to buy with the coins you have, and then decides which owned items to use on this turn's training or before a race. The whole policy is built around one idea: the climax/finale races at turns 74, 76 and 78 are where stat-boost items pay off most, and the summer training windows are where stat gains are largest, so the bot deliberately banks certain items for those moments instead of spending them on ordinary mid-career turns. All of this lives in career_bot/items.py; the user-facing knobs are in the SCENARIO OVERRIDES modal.

There is no manual "use this item now" button — item handling is fully automatic. What you control is the policy: reserves, thresholds, and which items the shop is allowed to buy. This section first walks through each item type (what it is, when the bot uses it, and how), then the strategy knobs that tune the whole system.

HOW THE SHOP RUNS

Each turn the bot reads your coin balance, ranks every item on offer into priority tiers, and buys top-down until it hits its coin floor for that turn. It will not buy a duplicate past that item's inventory cap (most items cap at 5; one-shot buffs/cures cap at 1). Buying spends down toward ~0 mid-career on purpose — leftover coins are unbought hammers and glow sticks, i.e. fewer fans.

The items, one by one

Cleat Hammers — Artisan vs MasterWhat: the single most important stat items. Artisan Cleat Hammer (25 coins) is the weaker tier; Master Cleat Hammer (40 coins) is the stronger tier. Both boost the payoff of a race. When/how: hammers are swung before races (handled on the pre-race path), and the bot follows a strict bank-for-the-finale policy. On a climax/finale race (turns 74 / 76 / 78) it always swings the best hammer it owns — Master first, then Artisan — no questions asked. This is the payoff the reserves exist for. On regular graded races the rules tighten once conservation engages at turn 25: it banks trackblazer_master_hammer_finale_reserve hammers (default 3, Master-preferred) for the three climax races and only spends surplus on regular G1/G2/G3. Artisans are always considered for a G1; G2/G3 respect their Artisan min-stock floors (defaults: G3 needs 3 in stock, G2 needs 2). Masters are never spent on G2/G3 — they are saved for the climax and G1s. Before turn 25 a G1 always swings the best hammer it has.
Megaphones — Coaching / Motivating / EmpoweringWhat: training boosters that stay active for several turns. Three tiers: Coaching Megaphone (40 coins, weakest), Motivating Megaphone (55), Empowering Megaphone (70, strongest). When/how: megaphones are used on training turns, not races, and only when the turn is worth it. Mid-career the bot fires them only during summer camp, on strong rainbow turns (2+ friendship partners on the chosen training), or when burning excess at career end — this stops the year-round small-gain misfires. It will not auto-fire the weak Coaching tier unless nothing better is owned. Strong megaphones are held back for the summer camps (it keeps a small summer reserve). In the late game the gates open up in stages: turns 60-64 (summer peak) fire on any non-zero training; turns 65-70 stop conserving and spend remaining stock; turns 71+ fire only on training that raises your current highest-priority non-capped stat.
Ankle Weights (anklets)What: per-stat training boosters — Speed / Stamina / Power / Guts Ankle Weights (50 coins each). When/how: an anklet only fires when the chosen training matches its stat (e.g. a Speed Ankle Weight on a Speed training) and the training's score clears a gain threshold (around 30, lowered while a megaphone is also active and during summer). It will not stack on top of an already-active anklet effect. Late-game windows mirror the megaphones: turns 60-64 and 65-70 fire on any positive matching training; turns 71+ require the highest-priority non-capped stat. The shop deliberately limits anklet stock (it keeps ~2 across all types) and weights buying toward the stat types your support deck actually trains, so it won't over-buy anklets you can't use.
Energy Drink MAX / Energy Drink MAX EXWhat: two very different items despite the similar name. Energy Drink MAX is a tiny +5 top-up whose only job is to escape 0 energy right before a race. Energy Drink MAX EX is a full +100 refill. When/how (the race-punishment recovery): the bot never spends an energy item before a race while energy is above 0. At exactly 0 energy it tops up only when the current race is the 2nd or 3rd in a consecutive-race chain (the 1st race's punishment is small, the 4th-and-later is capped), and it picks the smallest sufficient item — the +5 Energy Drink MAX first, then the smallest Vita. Past turn 70 there is no pre-race energy management at all, because post-70 races cost no energy and carry no 0-energy penalty. The full +100 EX is also used on training turns when energy is genuinely low (never over the vital cap). EX is on the always-exclude buy list by default, so the bot won't buy it unless you allow wasteful consumables.
Glow Sticks — fan multiplierWhat: Glow Sticks (15 coins) boost the fan reward of a race. When/how: used before G1/G2/G3 races whose fan reward clears Glow Stick Minimum Fans (default 20,000). Before conservation (turn 25) it uses them on any qualifying race. After that it banks trackblazer_glow_stick_final_reserve (default 1) for the final race and only spends surplus on big-fan races, but it always uses one on a top-tier G1 (30,000+ fans) and on the final race. The old x2 fan fudge is gone — the gate now compares the race's real fan figure directly (multiplier defaults to 1.0).
Cupcakes — Plain vs Berry Sweet (mood)What: mood/motivation items. Plain Cupcake (30 coins) and Berry Sweet Cupcake (55). When/how: the two are used for different jobs. Berry Sweet is the dedicated mood resource — spent to lift motivation when it drops below GOOD and energy isn't already comfortable. Plain is the kale-offset patch: Royal Kale Juice costs motivation, so when the bot drinks kale it spends a Plain Cupcake to cancel that mood hit (and it will refuse to burn a Berry Sweet on a kale offset). The cupcake+kale combo is locked to the summer turns mid-career. Past turn 59 the Plain reserve is dropped and any cupcake is spent for mood whenever motivation is below GOOD; in late-game dump mode it spends cupcakes up to GREAT so none are stranded. The shop stops buying cupcakes once it holds two (one in senior+), or when motivation is already maxed.
Vita 20 / 40 / 65 & Royal Kale Juice (energy)What: energy recovery. Vita 20 (+20, 35 coins), Vita 40 (+40, 55), Vita 65 (+65, 75), and Royal Kale Juice (a full ~100 refill, 70 coins, but it costs motivation). When/how: on a training turn the bot tops up energy when the gap to full is at least ~20 and energy is below Energy Threshold to use Energy Items (default 40). It picks Vitas greedily (largest that fits) without overshooting ~110% of max vital, and keeps Energy Item Emergency Reserve (default 1) untouched mid-career. Royal Kale is the panic lever: used at critical HP, or in summer when a Plain Cupcake can patch the mood loss. It does not spend energy items on a turn protected by a Good-Luck Charm (failure is already zero), and not on year-end race turns where the energy can never be used. Late-summer (60-64) and 65+ it spends Vitas freely (no reserve).
Stat Manuals / Notepads / ScrollsWhat: instant permanent stat gains. Per stat there is a Notepad (+3, 10 coins), a Manual (+7, 15 coins) and a Scroll (+15, 30 coins). When/how: applied instantly when owned. The bot buys Manuals and Scrolls early (they stabilise first-place finishes, which protects the coin economy) but skips the small +3 Notepads by default — they aren't worth the coins. Notepads are only bought when you are flush with coins (above the flush threshold ~250), so surplus that would otherwise sit unspent at career end converts into a permanent +3 instead. You can force notepad buying on with trackblazer_buy_notepads.
Mood / one-shot buffs & curesWhat & how: several support items the bot manages quietly. One-time buffs (Pretty Mirror, Scholar's Hat, Reporter's Binoculars, Master Practice Guide) are used once each as soon as owned. Reset Whistle (20 coins) re-rolls a lackluster training turn — held for summer camp and turn 65+ (where it only re-rolls if the best training's total gain is below the lackluster threshold, default 30). Good-Luck Charm (40 coins) guarantees a risky training succeeds — fired when failure rate clears its threshold (default 20%) and the main-stat gain is worthwhile. Cures (Fluffy Pillow, Pocket Planner, Rich Hand Cream, Smart Scale, Aroma Diffuser, Practice Drills DVD, and the catch-all Miracle Cure) are bought and used only when their matching bad condition is active — the bot does not pre-stock cures unless you turn on Preemptive Cure Reserve. Reporter's Binoculars, Master Practice Guide, Yummy Cat Food and Energy Drink MAX EX are on the always-exclude buy list (wasteful pre-race auto-consumables) unless you enable allow_wasteful_consumables.

These are set in the preset's mant_config directly (they have no v3 control). They are the highest-leverage late-game switches, so they are documented here even though you edit them in the preset rather than a modal.

When OFF (the default), past turn ~64 the bot enters dump mode: it stops conserving and aggressively spends every remaining item before the career ends — megaphones/anklets on any positive training, charms on any useful training (failure floor of 1), cupcakes up to GREAT mood, leftover energy/kale as recovery instead of resting, and it stops hoarding coins (keeps only ~60 for a turn-71 Master Cleat Hammer instead of the full 150 finale reserve). Set it to true to keep the conservative reserves running all the way to the finale instead.

(Automatically disabled past turn 64 in dump mode, since the goal then is to empty the inventory.)

The summer windows & the finale — why items are banked

Two facts drive the whole strategy. First, the summer training turns (37-40 and 61-64) are the main real stat source in Trackblazer — facility levels and gains peak there — so the bot deliberately saves megaphones, anklets, strong cupcakes and Vitas to spend during summer rather than dribbling them away earlier. Second, the climax/finale races at turns 74 / 76 / 78 are where Cleat Hammers and Glow Sticks pay off most, which is why hammers and a glow stick are banked for them via the finale reserves above.

FINALE RACES PAY NO COINS

The finale races award no shop coins. There is nothing to save coins for after them, which is why dump mode (the default) drains the inventory and the coin hoard in the late game — anything left unspent at career end is wasted. If you'd rather the bot stay conservative all the way through, set save_items_lategame: true in the preset.

TUNING TIP

If runs end with leftover hammers, lower the Master Cleat Hammer Finale Reserve and the Artisan min-stock sliders so more get spent on mid-career G1s. If the bot arrives at the climax with no hammers, raise them. If glow sticks sit unused, lower Glow Stick Minimum Fans so they fire on more races.

Per-trainee tuning under your preset

Character profiles

What a character profile is

A character profile is a per-trainee tuning layer that sits underneath your preset. Your preset (Training Settings, Racing Settings, etc.) holds your global settings; the profile lets one specific trainee carry its own race-solver tweaks and epithet goals without you having to reconfigure everything for every other trainee. Profiles are resolved automatically at the start of each run from the trainee's character ID (verified in career_bot/character_profiles.py).

Each profile is a read-only JSON file under data/character_profiles/. When a trainee has no hand-curated file, the bot auto-derives a sensible profile from that trainee's live aptitudes, so every trainee gets a profile, not just the curated ones. The resolution order is: hand-curated profile → auto-derived profile → default.json → empty.

What a profile actually controls

The character profile feeds three things into the Smart Race Solver (it does not override your turn-by-turn training picks — in v2.1 the engine's own training scoring is authoritative, confirmed in career_bot/runner.py):

Solver weight overridesA subset of the solver's scoring weights tuned for that character, layered on top of the global weights from Smart Race Solver Settings.
Target / Forced epithets (epithet goals)The race accolades the solver will route the schedule toward. Target epithets are a soft bias (they tilt the route toward the required races). Forced epithets are hard constraints (every selected forced epithet must have at least one matching scheduled race). This is the main lever and it routes the schedule by adding/keeping the races those epithets require.
Preferred distancesThe distances the solver should favor for that trainee, fed into the route alongside your global Distance Preference Mode.

How to view and edit a profile

There is no separate "profile editor" page; you tune the active trainee's profile through the controls that consume it:

  1. In Setup, pick the trainee from the Library. That selection is what the profile is resolved from.
  2. Open Smart Race Solver Settings (the gear on the Trackblazer card). Use the Character Preset list and REFRESH PROFILE to confirm the active character.
  3. Scroll to TARGET EPITHETS and FORCED EPITHETS. Click epithet cards to select goals, or use the search box. These selections are the profile's epithet goals and are persisted with the solver config (endpoints /api/smart-solver/config and /api/character-profile/epithets).
  4. Adjust Distance Preference Mode and the per-distance aptitudes in the same modal to shape how the goals get routed.

Auto-pick signature epithets

Each known character has a "signature" epithet. The profile can auto-seed it as a soft target when you have not set any target epithets yourself. This is opt-in and OFF by default (default flipped to off in v6.7.6, verified in career_bot/character_profiles.py and the /api/character-profile/auto-pick endpoint). It is off by default on purpose: the solver already wins high-value races organically, and forcing a signature can bias the route away from its natural fan-maximizing path and drop other valuable races. The precedence chain is your explicit targets > profile JSON > auto-pick.

When to edit a profile

WHEN TO TOUCH THIS

Only edit a profile when you want one trainee to behave differently from your global settings — e.g. you want this run to chase a specific epithet/title, lock a distance, or nudge the solver's weights for a quirky aptitude spread. If you just want the bot to win as many fans/races as possible, leave the defaults alone: the auto-derived profile and your preset already handle it, and an over-eager forced epithet can make the schedule worse, not better.

TIP

Prefer Target epithets over Forced for casual goals. Target is a soft bias the solver can balance against fans; Forced is a hard requirement that can crowd out other races. Set-bonus chasing across all achievable titles is a separate global toggle (Chase Achievable Set-Bonuses (Epithets)) in Smart Race Solver Settings.

The DASHBOARD is mission control once a career is running. The CAREER pill — status and delete control

Action Log — what the bot did each turn

The ACTION LOG is the turn-by-turn ledger of the career. Every poll, new turns are accumulated into an archive (earlier turns are never dropped), so you can scroll back through the entire run. The header reads e.g. 41 TURNS · LIVE (or · IDLE when stopped). Each row has five columns:

The log follows the live tail: as long as you're scrolled to the bottom it stays pinned to the newest turn, but if you scroll up to read history it leaves your scroll position alone so new rows don't yank you down.

Decision Reasoning — why it chose that

To the right of the log, the DECISION REASONING panel explains each turn's choice as a stack of cards (newest at the bottom; older cards fade out so the current decision stands out). Each card carries:

  • A title like T41 · SPEED TRAINING plus a colored action badge matching the log.
  • A category sub-label describing which subsystem made the call — GOAL-AWARE SCORER (training), SMART RACE SOLVER (racing), ENERGY GUARD (rest), MOOD GUARD (recreation), or INFIRMARY.
  • The engine's reasoning as bullet points when there are multiple lines, or a single sentence when there's just one. These come straight from the engine's per-turn reasoning; if the engine didn't emit any for that turn, a sensible generic sentence is shown instead.
  • An ITEMS row of chips when items were bought or used that turn. Shop items are shown with their icon (e.g. a megaphone, cleat hammer, cupcake), and any item name mentioned inside the reasoning text gets its icon inlined too.

Like the log, the reasoning panel tail-follows: pinned to the newest card while you're at the bottom, frozen in place while you read back. The single DECISION REASONING tab is the only view here.

TIP

Read the log and the reasoning side by side. The log tells you what happened (e.g. “RACE · Tenno Sho”); the matching reasoning card tells you the engine's justification (e.g. why the Smart Race Solver entered that race, or why it rested to protect HP before a cluster).

These differ from the settings modals: they're runtime knobs that take effect immediately (or on the next run), and the on/off toggles persist across page navigation via the browser. Tempt Fate — bot speed & turn delay

The TEMPT FATE button (gear icon) opens a BOT SPEED popover. "Tempting fate" means trading anti-detection pacing for raw speed — the faster levels strip out the human-like delays. There are four speed cards (verified against the backend SPEED_PRESETS in main.py):

SafeFull human-like delay using your custom range (below). Keeps the inter-turn pause and full per-call API timing (api_scale 1.0). The safest against detection — recommended default.

Custom Delay (S) — the Safe-mode pause range

Below the speed cards the popover has a CUSTOM DELAY (S) section with two number fields — a min and a max, shown as min – max (defaults 2.5 and 5 seconds). This is the randomized inter-turn pause range the bot uses in Safe mode: each turn it waits a random number of seconds inside this window before acting, which is what makes Safe mode look human. To use it, type a number into either field and click away (the field commits on change) — the new range is pushed live via POST /api/settings/turn-delay. Widen the range (e.g. 4 – 9) for a more conservative, less regular cadence; tighten it for a steadier pace. The non-Safe speed levels send this delay as disabled, so the min/max fields only matter while Safe is selected.

  1. Click TEMPT FATE to open the popover.
  2. Click a speed card — it applies live via POST /api/settings/speed, so a running career picks it up immediately.
  3. For Safe, edit the CUSTOM DELAY (S) min/max fields; the range is pushed via POST /api/settings/turn-delay. (The non-Safe levels send the delay as disabled.)

The button glows "on" whenever the speed is anything other than Safe, and it re-syncs to the real server speed after you navigate between pages, so it never shows a stale state.

CAUTION

The faster the level, the less the bot mimics human timing. For overnight unattended farming the safer levels reduce the chance of tripping anti-bot detection. Ludicrous removes pacing entirely.

Burn Clocks

The BURN toggle (shows BURN:ON / BURN:OFF) lets the bot spend clocks (the race-retry currency) to retry races it didn't win well enough. With it off, the bot accepts race results as-is and conserves clocks.

  • Click BURN to flip it; the state persists across pages.
  • It is a true live runner toggle, not just a start-payload flag: flipping it fires POST /api/career/runner/burn_clocks immediately, so a career that's already mid-run starts (or stops) burning clocks on its very next race — you do not have to restart the run. The same value is also folded into the start payload for fresh careers.
  • What actually gets spent (clocks vs carats, caps, which races) is governed by the RETRIES popover below.

Event choices

During a career the game throws up story events (from your trainee and her support cards) that ask you to pick one of two or more options. The EVENT CHOICES page (the events.html tab) is where you inspect every event the bot knows about, see what each option does, and force a specific option when you don't want the bot to decide for itself. By default the bot answers every event on Auto, and Auto is already a stat-aware decision rather than a coin flip.

How Auto scoring works

When an event appears, the engine (career_bot/events.py, EventManager.choose) walks a fixed precedence ladder and stops at the first rule that applies:

  1. Your override wins first. If you've forced a choice for this event (in the active preset, or the legacy runtime file), that option is taken immediately, no scoring.
  2. Hardcoded-critical events. A small number of events have a known-correct answer baked in (for example story id 400004002 always takes option 3). These are taken verbatim so a critical story branch is never fumbled.
  3. Scored against effects. Otherwise the bot reads each option's actual effects and scores them. Effects come first from any reward data carried in the live event payload, then fall back to the gametora + master.mdb effects database (the "Skill +20, Bond +5, Energy +10" strings). Each effect is weighted using your Event Choice Prioritization stat order, your energy/mood/bond/skill-hint preferences, the current turn (energy is worth more in summer camp and the finale lead-in), and how close a stat is to the 1200 cap. The highest-scoring option is picked.
  4. Safe fallback. If an event has no usable effect data at all, the bot avoids the blind first-option bias: it takes the second option when there are multiple, otherwise the only option. This is the conservative default the original bot used.

Every decision leaves a breadcrumb (the reason shows up as preset_override, hardcoded_special, weighted_outcome, or fallback_second_if_possible in the bot's trace), so Auto is never a black box.

TIP

The Event Choice Prioritization stat order lives in TRAINING SETTINGS → PRIORITIES. Reorder it (e.g. Speed before Wit) and Auto's event scoring will start preferring options that give those stats. You don't have to force individual events to bias Auto.

MODE column (AUTO / FORCED)The right-hand badge: a neutral AUTO chip means the bot scores it; an amber FORCED badge means you've pinned an option. Read-only here — change it from the detail panel.

Click any list row to open EVENT DETAIL on the right. It shows the category badge, the effect source when known (e.g. src game8), the event name, the seen N× this profile count, and then every option as a Choice 1 / Choice 2 … card with its recorded effect chips (green for gains, red for losses, grey no recorded effect when nothing was parsed), followed by the OVERRIDE MODE toggle and the note "Override is saved to the active preset and applied at the next career start."

How to force (override) a choice

  1. Open the EVENT CHOICES tab and find the event (use the search events box, or the Story/Support tabs).
  2. Click the event row to open EVENT DETAIL.
  3. On the option you want, click FORCE THIS. The card gets an amber FORCED ✓ badge, the list row flips to a FORCED badge, and the override counter increments.
  4. To go back to letting the bot decide, switch OVERRIDE MODE at the bottom of the detail panel to AUTO (the FORCE segment re-pins the current option, or Choice 1 if none was set).

The override you set is the option position (Choice 1 = position 0 internally); the page translates that to the game's real select index for you, so you always think in terms of "Choice 1, Choice 2".

Where overrides are saved and when they apply

Overrides are stored on the active preset (POST /api/events/override with {story_id, choice} where choice is the 0-based position, or -1 for Auto → preset_store.save_event_overrides). That means:

  • They persist across restarts and across careers — they live in the preset, not in memory.
  • They are per-preset: switching to a different preset switches its event answers, so you can keep one set of choices per strategy.
  • They take effect at the next career start (the runner reads preset.event_overrides when a career begins), as the detail panel notes: "Override is saved to the active preset and applied at the next career start." A mid-career force won't retroactively change an event already shown earlier in the same run.

To wipe every override at once, click CLEAR ALL OVERRIDES in the header — this resets every event back to Auto in one action (POST /api/events/overrides/clear).

The event database

The catalog is populated even before you've ever run a career: the bot ships an effects database scraped from gametora.com with skill names resolved via the game's master.mdb (data/event_effects.json, covering support-card and trainee/scenario events), so you can pre-set overrides up front. Once a career runs, events you actually encounter are merged in from uma_runtime/events_seen.json with their real seen counts and option indices. gametora + master.mdb is now the sole effect/outcome source (the old game8 scrape and the curated outcome knowledge base were removed), so the effect chips you see are exactly what the scorer uses, and the detail panel labels them src gametora.

GOOD TO KNOW

An event whose options read no recorded effect simply means the database has no parsed effect string for it. On Auto it will use the safe fallback (second option). If you know the right answer, force it explicitly so it isn't left to the fallback. Likewise, an event still showing seen 0× is a catalog entry you haven't hit on this profile yet — you can pre-force it now and it'll be honored the first time it appears.

Accounts & multi-account

Icarus signs into Umamusume through Steam, and it can supervise several accounts at once, each running as its own bot instance on its own port. This section covers how login and saved-session reconnect work, and how the MULTI-ACCOUNT MANAGER page (the accounts.html tab) monitors a fleet.

Steam login and saved-session reconnect

The first login needs a fresh in-game auth capture plus a Steam session ticket. The bot console waits for you to launch the game and enter the menu so it can capture the live auth fields (viewer_id, udid, a 48-char auth_key, app/resource versions), then it gets a Steam ticket from your Steam username/password (and a Steam Guard code if Steam asks). On success (POST /api/login) the bot validates the capture (has_fresh_auth_config) and saves the whole session.

After that first login, reconnects are headless — you normally don't have to re-enter anything:

  1. On startup the bot calls check_saved_auth() and tries the saved session ticket directly (the "headless bypass").
  2. If the saved ticket has expired, it transparently re-derives a new Steam ticket from the saved Steam credentials and retries.
  3. If a ticket is refreshed mid-run, it's written back to the saved config so the next run still works (_persist_refreshed_ticket).
STEAM GUARD

If Steam asks for a Steam Guard code, enter the current one (codes refresh every 30 seconds). The bot turns raw Steam errors into plain guidance: a rejected code or password tells you exactly what to fix, and a rate-limit error tells you to wait ~15–30 minutes before retrying (friendly_steam_login_error). If a Guard code is rejected on the very first attempt, wait for a fresh code and try once more.

How your credentials are stored (credential safety)

Your saved auth lives in your userdata folder (userdata/auth/<profile>/auth_config.json and steam_token.txt), not inside the app build folder. That placement is deliberate: the build folder is wiped on every version upgrade, so keeping auth in userdata means the headless reconnect survives upgrades.

  • Your Steam username and password seed are obfuscated before being written (base64 of the reversed string, with an enc: prefix) and de-obfuscated only in memory at login — they are not stored in plaintext (_obfuscate_creds / _deobfuscate_creds).
  • Auth is saved per profile, so different profiles keep separate sessions.
  • Hardware identity is spoofed per-profile for separation, but the capture's udid/device_id are never overwritten because that would break the auth crypto binding.
NOTE

Obfuscation is light protection against casual reading of the file, not strong encryption. Anyone with access to your userdata folder could still recover the credentials, so protect that folder like any other saved login.

The actual supervisor is manager.py. When running it launches one child bot per roster entry, health-checks each on a loop, writes a live manager_status.json the page reads, and applies each account's restart policy:

auto_restartWhen true, a crashed or stuck child is restarted automatically (with exponential backoff between attempts).

The backend exposes GET/POST /api/accounts to read and write it, and POST /api/accounts/manager/start to launch the supervisor (passing your userdata path so the manager reads the exact same roster the dashboard wrote).

AI / Misc — the three learning layers

Icarus has an entirely local, offline learning system. Nothing is sent to the cloud: every model, table and report is built from your own completed careers and written to JSON files under the runtime folder (the uma_runtime/ai/ directory). You open it from the navbar tab DIAG / AI, then the AI & MISC segment at the top. It is organized as three independent layers, each more advanced (and more optional) than the last.

THE BIG PICTURE

Layer 1 (outcome risk) is always on and needs no setup. Layer 2 (Shadow Mode + Live Policy Assistance) trains a small local model in the background and only starts changing decisions if you explicitly enable it. Layer 3 (local LLM advisor) is a chat model you host yourself that only ever writes advice — it never touches gameplay. The deterministic engine is always the final authority.

Reading the AI dashboard (the four KPI boxes + header)

The top of the page is a strip of four live stat boxes (built in diag.js aiView, fed by /api/ai/dashboard and /api/ai/status). They summarize how much the model has learned and how good it currently is. None of them are settings — they are read-outs you watch climb as you complete more careers.

DATASETTotal turn decisions collected — every train/race/rest/recreation choice from your career logs that has been exported into the AI training set (status.turn_decisions.rows). This is the raw fuel for everything else; it grows automatically as you run careers and after REBUILD DATASET / IMPORT LOGS. The same number, plus the model version, is also echoed under the page title (e.g. v2 · 12.4k samples).
ACCURACYModel precision as a percentage (the shadow report's precision, shadow_mode.precision). It answers "when the learned model warned about a race, how often was that warning correct?" Higher is better; this is the number you watch before trusting Layer 2.
LAST TRAINHow long ago the model was last (re)built, shown as a relative time (e.g. 14m ago), with an auto-training sub-label. Sourced from the dashboard's updated_at / the trainer's last run timestamp. If it says — the model has never been trained.
SHADOW WINThe Shadow-Mode precision (vs live policy) — the same precision figure framed as "how the shadow model would have scored against your real outcomes." This is the headline gate for enabling Live Policy Assistance: you want it past 60%.

Two more read-outs sit nearby: the TRAINING PIPELINE card header shows an auto-training ON/OFF chip (reflecting whether the background trainer is enabled), and the page title area shows a model healthy / untrained status plus the model version string (currently v2, from AI_TRAINER_VERSION in ai_trainer.py). An untrained model shows an amber banner instead of metrics — the dataset count and system health below are still live.

Layer 1 — Outcome Risk (statistical, on by default)

Every time a race finishes, Icarus records the result into a small per-program tally (career_bot/race_intelligence.py record_race_outcome, called from runner.py). For each race program it tracks starts, wins, losses, average finishing rank, clean wins, and wins that only happened after a clock retry. When the Smart Race Solver later scores that same race, it looks up this history (race_outcome_risk) and applies a penalty to races you keep losing or keep barely scraping with clocks — steering future careers toward races you actually win.

This is purely statistical (no machine-learning model, no training step). It is grouped two ways: a profile bucket keyed to the exact preset + trainee, and a broader program bucket across all your runs; the profile bucket wins once it has enough samples. Confidence rises with sample count: 2 starts is treated as low, 5+ as high.

Enable Outcome RiskMaster on/off for layer 1. Default ON (verified career_bot/trackblazer.py: enableOutcomeRisk defaults to True). When on, the solver subtracts an observed-risk penalty from races with a poor local track record.
Outcome Risk WeightMultiplier on that penalty (outcomeRiskWeight, default 1.0, clamped 0–5). Raise it to avoid problem races more aggressively; lower it toward 0 to nearly ignore history. Set in mant_config as outcome_risk_weight / enable_outcome_risk.
Clock-dependency riskIf you race with Burn (clock retries) off, the solver adds an extra penalty to races you historically only won by spending clocks (clockDependencyRiskWeight) — because without clocks those would have been losses. When Burn is on, this term is skipped.

How to use it: do nothing — it just works and gets smarter the more careers you run. The only knobs are the weights above, exposed via mant_config. There is no minimum: with zero history every race scores neutrally, so a brand-new install behaves exactly as if the layer were off.

Layer 2 — local model: Shadow Mode & Live Policy Assistance

After enough completed careers, Icarus can build a richer learned model from your AI dataset. This is the TRAINING PIPELINE card on the AI page. It turns your raw turn-decision logs into analytics tables and confidence-gated policy adjustments for races, items and event choices (career_bot/ai_trainer.py). The four KPI boxes above (DATASET, ACCURACY, LAST TRAIN, SHADOW WIN) all describe this model.

Building / refreshing the model. The TRAINING PIPELINE card is a row of four buttons (left to right). Each maps to an offline maintenance endpoint under /api/ai/* — none of them touch a live career while it is running.

REBUILD DATASETRe-exports the AI-ready JSONL training files from your existing career logs (POST /api/ai/rebuild-dataset). Offline maintenance only — it never changes live play. Use it when the DATASET count looks stale, or right after you copy in old logs, to refresh the training set without retraining the model.
IMPORT LOGSPulls career logs (and optionally presets) from an older Icarus build, a runtime folder, or a zip, so a fresh install starts with all your accumulated history (POST /api/ai/import-logs). You paste a Windows path such as C:/OldBuild or a .zip path. Gameplay logs, AI-safe aggregates and (optionally) presets are imported; auth files are deliberately ignored. It can rebuild the dataset and train in the same pass.
TRAIN NOWBuilds everything in one pass (POST /api/ai/train-now): the analytics tables, the learned race/item/event models, the policy adjustments, the Shadow-Mode report and the backtest. Use it after importing old logs or whenever you want the KPIs refreshed immediately rather than waiting for the timer. The same run also happens automatically after a career completes and on the auto-training interval.

FIRST-TIME / FRESH-INSTALL FLOW

Coming from an older build? 1) Click IMPORT LOGS and point it at your old runtime/build folder or zip. 2) Click REBUILD DATASET if the DATASET count didn't jump. 3) Click TRAIN NOW to build the model and populate the KPIs. After that, just play — auto-training keeps everything current.

Auto-training. The header chip auto-training ON/OFF reflects the background trainer. By default it is enabled and retrains after each completed career plus on a timer (interval_minutes, default 60), but it skips while a career is actively running (so it never competes with live play). All of this is configurable through /api/ai/auto-training/config.

Shadow Mode (proof before you trust it). Shadow Mode is on by default and is the safety gate for the whole layer. It replays the learned race-risk hints against your real historical outcomes without changing any live decision: a hint that would have penalized a race you went on to lose counts as a useful warning; a hint that would have penalized a race you actually won counts as a false alarm. The ratio is the model's precision — shown as SHADOW WIN / ACCURACY. This lets you see whether the model is actually right before letting it touch anything.

Live Policy Assistance (LPA) — the only part that changes decisions. When you turn LPA on, the learned adjustments are allowed to nudge the scores of races/items/events the solver was already legally considering. It is deliberately conservative and reversible: it never invents an action, never overrides a deterministic gate, and only applies adjustments above a confidence threshold. It is OFF by default (verified ai_trainer.py: enable_live_policy_assistance: False).

Enable Live Policy AssistanceMaster switch for LPA. Default OFF. When on, high-confidence learned hints adjust candidate scores; the engine still chooses and executes. Mapped in the engine as enableLiveAiPolicy.
Confidence thresholdMinimum model confidence before an adjustment counts (default 0.65). Hints below this are ignored. Engine key liveAiConfidenceThreshold.
Max live adjustmentCaps how many score points a single learned hint can move a candidate (default 25), so no one race can be massively over- or under-weighted.
Win-rate ceilingSuppresses risk warnings on races you usually win anyway (default 0.50). This stops the model from flagging races that then win, which previously dragged shadow precision down to ~16%.
WHEN TO ENABLE LPA

Only enable Live Policy Assistance once the model recommends it. The page surfaces a recommendation that flips to ENABLE only after all of these are true: roughly 250+ turn records, race-result coverage ≥ 85%, at least 50 race rows, at least one learned adjustment above the confidence gate, and — once Shadow Mode has evaluated 100+ hints — shadow precision at or above 60%. If the recommendation still says KEEP DISABLED, leave it off: enabling a low-precision model suppresses more good races than bad ones. In short: run many careers, watch SHADOW WIN climb past 60%, then enable.

Other AI cards on this page. The ADVISOR · LATEST card shows the most recent deterministic post-run note (/api/ai/advisor/latest) and carries two buttons:

VIEW POST-RUNOpens the full post-run report for your latest career (GET /api/ai/post-run/latest) — the complete write-up behind the one-line advisor headline, including the risky-race list and suggested config tuning. Use it after a disappointing run to see exactly which races/decisions the model flags.
BACKTESTShows the latest backtest report (GET /api/ai/backtest/latest): how often the current model would have flagged your historically-bad races, plus the late-Senior race count it checked. This is your sanity check that the model catches real problems before you ever let LPA act.

Layer 3 — Local LLM Advisor (LM Studio / Ollama)

A debugging playbook

  1. Reproduce the problem, then come straight to DIAGNOSTICS.
  2. Scan SYSTEM HEALTH top to bottom: a red FastAPI server means restart the backend; an unauthed Steam bridge means fix login/auth; a bad Userdata path means your settings aren't where you think; stale Master data means re-sync.
  3. Switch the CONSOLE to ERROR and read the first failing line — it usually names the culprit subsystem. If a run died outright, click CRASH TRACE for the exact exception.
  4. If you can't resolve it, click DEBUG BUNDLE (or the share-safe AI bundle for learning issues) and attach the zip to your report. Use EXPORT LOGS instead when the maintainer only needs your career history.
STALE-SERVER GOTCHA

If you edited config or code and a fix "isn't working," the running server may be stale: the Python backend loads its modules once at startup. A browser reload is not enough — fully restart with python main.py so on-disk changes take effect. Use the FastAPI server Python-version line and LAST TRAIN timestamps to confirm you're talking to a freshly started process.

Strategy & tips

Icarus executes a strategy flawlessly, but it can only work with the build you give it. These are the gameplay realities that decide what a run can actually achieve — several of them are levers outside the bot. Internalize these before you start blaming a setting.

Parent sparks (inheritance) are the real stat lever

The two umamusume you inherit at career start grant blue stat sparks (and pink aptitude sparks, green unique sparks, gold scenario sparks) that are completely separate from your support deck. A trainee with strong stat sparks starts far ahead and needs fewer training turns to reach high stats — which frees turns to race more. If a run underperforms a setup you use elsewhere with the same deck, the difference is almost always sparks, not a bot setting.

  • On the Setup page, hover any PARENT 1 / PARENT 2 / guest-parent card to pop the SPARKS tooltip — it shows the parent and both grandparents with each factor colour-coded (blue = stat, pink = aptitude, green = unique, gold = scenario) and its star count, plus G1/G2/G3 win chips.
  • Use the parent search + factor filter on the PARENTS / GUEST PARENTS tabs to find lineages stacked with the stat sparks your trainee needs.
  • For both high stats and high fans, strong parent sparks are usually the missing ingredient — there is no setting that substitutes for them.
WHERE THE SOLVER SEES THEM

In Smart Race Solver Settings → Aptitudes, each aptitude row reads Base · Manual Start · Sparks +N · Solver Final. The estimated parent sparks are layered on top of the base grade to produce the Solver Final grade the route planner actually uses — so good aptitude sparks also widen the pool of races the trainee is eligible for.

Race count vs. stats is a hard tradeoff

A Trackblazer career is 78 turns. Every race consumes a turn that could have been training — you cannot maximize both fans and stats. Decide up front whether a run chases fans (race more, lower stats) or stats (race less, push targets). Around 30–41 races is the typical healthy range for a Mile/Medium trainee.

  • Max Streak (the SCORING WEIGHTS row in the Smart Race Solver, default 4) is the single biggest lever on total race count. For a typical trainee, Max Streak 5 schedules ~37 races and 8 reaches ~41. Set it to the number you want, then let the solver fill the calendar.
  • The OPTIMIZATION WEIGHT PRESET macro gives you two one-click starting points: Fans + Epithets (fan-leaning, the default) and Stat Epithets (stat-leaning). They just rewrite the scoring weights below; tune from there.

Over-racing is intentional in this bot

You may notice Icarus enters more races than a hand-played run would. That is by design — fans are the primary farming metric and a slightly over-raced schedule banks more of them. Do not try to "fix" it by slashing Max Streak to the floor unless you are deliberately optimizing for stats over fans. The energy/rest logic still protects you from racing into a loss (see below).

Summer is the main stat source — protect it

The two summer camps (turns 37–40 and 61–64) run at Lv5 facilities and are where the bulk of your real stat growth happens. The engine deliberately trains all-out through summer and the finale (it banks stats and avoids resting on those turns), and the item layer saves its best consumables for these windows because finale races pay zero coins. Two consequences for how you configure things:

  • Leave Must Rest before Summer on (Training Settings, default ON) so the trainee enters camp with full energy and gets the most out of every camp turn.
  • Don't force voluntary races into the summer turns. In the Smart Race Solver, Allow racing during Summer (Classic / Senior) is OFF by default for exactly this reason — only turn it on if a high-value target genuinely lands in a camp turn.
  • After turn 64 the item manager flips to a "dump" posture (it stops conserving and spends remaining items on any slightly-useful training), because summer is over and finale races give no coins. The save_items_lategame behaviour controls this; the default is to dump.

Match the deck and sparks to the trainee

Align your six-card support deck, your friend support, and your parent sparks with the trainee's preferred distances and running style. Rainbow (friendship) training depends on which cards appear together, so the deck shapes how fast each stat climbs. Use Recommended Support Cards on the Setup page — it shows the scraped Game8 Trackblazer build for the trainee (with the cards you own marked) or, if there is none, the best picks from your owned collection. A recovery (gold) skill is often worth more than raw stamina for stamina-hungry builds, because it effectively lowers the stamina needed to clear races.

DON'T MICRO-MANAGE THESE

Grade-point quota and in-game race-bonus auto-complete or are soft inputs — you don't need to optimize them. The global stat cap is 1200 for every stat, and fatigue stops accumulating after Late December, which is why the finale window trains flat-out. The defaults already account for all of this.

Glossary

Every term you'll see across the dashboard, Setup, and settings, defined. Resource terms map to the live pills in the status rail; scenario terms map to the Trackblazer scenario the bot runs by default.

Resources & currencies

TPTraining Points. The energy currency spent to start and continue a career. Shown in the rail as current/max. When it runs low the bot restores it per your TP RECOVERY setting (items, carrots, or both) before training.
CarrotsThe bulk recovery currency used to refill TP. Shown as the amber CARROTS pill. One of the two TP-recovery sources.
GoldPremium currency, shown compacted (e.g. 1.2M) in the GOLD pill. Not spent by routine training.
ClocksUsed to retry a race you lost. Free continues always fire automatically; spending paid clocks on a retry requires BURN to be ON. The CLOCKS pill shows your stock. The Retries popover caps how many paid clocks a career may spend.
CaratsA secondary retry currency. With Use carats when clocks run out enabled (Retries popover), the bot falls back to carats for retries once its clock budget is exhausted.
CoinsThe in-scenario shop currency the item manager spends on consumables. The bot holds a finale-aware coin reserve (tunable via Coin Reserve Override); note finale races pay zero coins, which is why items are dumped after turn 64.

Inheritance & outcomes

Sparks / factorsInheritable bonuses carried by a finished trainee and her ancestors. Categories: stat (blue, starting-stat bonus), aptitude (pink, raises distance/surface grades), unique (green, a unique skill), scenario (gold), plus skill/race factors. Each has a star rating (1–3). Parent sparks are the biggest off-deck stat lever — see Strategy & tips.
Career gradeThe overall final grade awarded for a completed career run (the trainee's evaluation rank).
FansEarned from races and other sources; the primary farming metric. The MONITOR dock logs per-turn fan gains and the live total appears in the dashboard's Lifetime Metrics.
EpithetA race accolade / title earned by meeting a condition (e.g. "win 3 turf G1 races", "reach 1200 Stamina"). The Smart Race Solver can chase them: Target Epithets bias the route toward the named races (only when set-bonus chasing is on); Forced Epithets are hard constraints that must each have a matching scheduled race. Completing a set/title can grant random-stat bonuses.
Milestone %How far toward your final stat targets the bot aims by each year-end. End of Classic Year Milestone (default 33%) and End of Senior Year Milestone (default 66%) pace the training push across the career.
Max StreakThe maximum number of races the solver will plan back-to-back (SCORING WEIGHTS row, default 4). The biggest single lever on total race count.

Scenario & mechanics

TrackblazerThe default scenario, "Start of the Climax". A 78-turn career with climax/finale races at turns 74, 76, and 78, a global stat cap of 1200 on every stat, two summer camps (turns 37–40 and 61–64), and fatigue that stops accumulating after Late December.
AptitudeThe trainee's letter grade (S–G) for each distance (Sprint / Mile / Medium / Long) and surface (Turf / Dirt), plus running style. Read live from the game during a career; you can override grades in the solver's Aptitudes section. The Aptitude Threshold sets the minimum grade (distance AND surface) for a race to be eligible.
Running styleHow the trainee paces a race: Front Runner, Pace Chaser, Late Surger, or End Closer (plus Auto). Set in Racing Settings, optionally per-distance or per-race.
Recommended skillsThe top-tier skills the skill buyer prioritizes for a trainee's style and distances. The Stop After Recommended Skills toggle decides whether buying halts once those are owned.
RainbowThe friendship/rainbow training multiplier you get when a bonded support card glows on the facility you train. The bot rewards rainbow turns; Attenuate Rainbow on Capped Stats scales the bonus down as a stat fills toward its cap (off = flat 2.0x).
BondYour friendship level with each support (orange → green → blue/rainbow at 80). Higher bond unlocks rainbow training. Training Settings has knobs for low-bond value, the bond-to-rainbow push, and overall bond priority weight.
MoodThe trainee's condition (Awful → Great). Higher mood improves training gains and lowers failure chance; the bot takes recreation or uses mood items when it dips and trades off poorly against training.
Group outingThe special recreation tied to Team Sirius and Heirs to the Throne group support cards. Used well it powers a trainee toward SS/UG. The Sirius Throne Strategy setting controls whether it fires on a fixed schedule, opportunistically, or never.
Outcome riskA statistical (no-model) record of how each race has historically gone for you. With Enable Outcome-Risk Avoidance on (default), the solver penalizes races you've lost before so it routes around them.

Troubleshooting

The fixes for the issues testers hit most. If something isn't covered here, open Diag / AI to confirm the bot is connected, then read the Decision Reasoning panel to see exactly what it decided and why — those two explain the large majority of surprises.

"OFFLINE" banner across the top

The banner reads OFFLINE · backend not reached. It means the page can't reach the bot, so no live data loads.

  1. Start the server: run python main.py.
  2. Reload the page. The banner clears automatically on the next poll (every ~1.5s) once it connects — you don't have to time the reload.

Resource pills show — (TP / Carrots / Gold / Clocks blank)

A dash means the Steam bridge is reachable but not authenticated, so account data can't be read.

  1. Open Accounts and press Login (or fill the Steam login overlay if it appears).
  2. On Windows, login frequently needs Icarus to run as administrator. If Login fails or the pills stay —, relaunch the app elevated and try again. Auth is saved to userdata and reused, so this is occasional, not every run.

A change isn't taking effect

This is the most common false alarm. Python loads modules once at startup, so an on-disk change to engine/item/skill behaviour does not apply until you fully restart the server.

  1. For backend changes (engine, items, skills, runner): stop and restart python main.py. A browser refresh only reloads the UI, not the running bot.
  2. For settings you changed in a modal: confirm you pressed SAVE — each settings modal POSTs to its endpoint on save and writes into the active preset. Closing without saving discards the change.
RULE OF THUMB

If a fix "isn't working" in a live run, restart the server first before assuming the fix is wrong — a stale long-lived server masks correct on-disk code.

My race count is too low (or stats fall short)

Race count is governed by Max Streak in the Smart Race Solver (SCORING WEIGHTS, default 4).

  1. Raise Max Streak toward 5–8 for more races, then re-solve.
  2. Consider enabling Include OP / Pre-OP races to widen the schedulable pool, and loosen Distance Preference Mode from Strict toward Balanced/Loose.
  3. Stats falling short of targets is expected when you race a lot — targets are aspirational guides, and parent sparks make up the difference. If you want higher stats, race less (lower Max Streak) and lean on the Stat Epithets weight preset.

Low win rate / raced at 0 energy

Racing at critical HP loses races. If you see the bot entering races on empty:

  1. In Racing Settings, turn OFF Ignore Low Energy Racing Block (it should be off by default) so voluntary racing is blocked at critical HP during a streak.
  2. Keep Ignore Consecutive Race Warning off so the race-chain safety break stays active.
  3. In Scenario Overrides → Energy & Resources, set a sensible Energy Threshold to use Energy Items (~40, the default) so the bot tops up before training rather than failing into it.

The bot isn't buying skills (SP piling up)

Two settings in Configure Skills usually cause hoarding:

  1. Skill Point Threshold too high — lower it so the bot buys steadily instead of waiting for a huge balance. Lots of unspent SP at career end is the tell.
  2. Stop After Recommended Skills ON — if your trainee has no top-tier "recommended" skills, this can discard every good candidate and buy nothing. Leave it OFF (the default) to keep buying good skills. The pre-finals dump still overrides it either way.
  3. Note: skills you already own are never re-bought — if the bot "skips" a skill it may simply already have it.

Steam says "rate limit" or rejects my 2FA code

This comes from Steam, not the bot — Steam throttles sign-ins after rapid or failed attempts.

  1. Wait ~15–30 minutes (the login button shows a cooldown countdown if Steam returned one).
  2. Try once with a fresh Steam Guard code — codes refresh every 30 seconds, so let a new one appear before submitting.
  3. If the overlay asks for a Steam Guard code (VALIDATE step), enter the current authenticator code exactly.
KNOWN QUIRK

Steam auth capture sometimes needs the login flow completed twice — the first attempt may be rejected and the second succeeds. If a single attempt fails, simply repeat it once before assuming something is broken.

2511The support deck had too many cards. Icarus trims it to a legal six automatically — if it persists, rebuild the deck in Build Custom Deck (5 of your own + 1 friend) and re-apply.

Guest parent expired

Guest (borrowed) parents are time-limited by the game. The career start uses one of your own parents plus one guest in the PARENT 2 · GUEST slot.

  1. On Setup, open the GUEST PARENTS library tab and press LOAD GUEST PARENTS (use the force-refresh to fetch a fresh list).
  2. Pick a current guest — it occupies the PARENT 2 slot — then start the career.
  3. The guest age filter (Created < 1h / 6h / 24h…) helps you pick recently-listed guests that won't expire mid-setup.
